开发良好习惯

1. 代码风格 - Linus Style

核心原则
简洁清晰：代码要让人一眼就能看懂在做什么
实用主义：不过度设计，解决实际问题
可维护性：6个月后的自己应该能看懂代码
命名准确：变量和函数名要准确描述其作用，不要含糊
具体实践
函数要短小精悍，一个函数做一件事
避免过深的嵌套（最多3层）
错误处理要明确，不要隐藏错误
注释解释"为什么"而不是"是什么"
优先使用标准库和成熟的工具
不要重复造轮子
2. 开发前的思考

每次动作前必须回答三个问题：
为什么要做这个？
这个功能解决什么问题？
它在整个系统中的位置和作用是什么？
不做会有什么影响？
当前项目情况如何？
现有代码结构是什么样的？
这个改动会影响哪些现有功能？
有没有可以复用的代码？
执行过程中的关键注意点？
哪些地方容易出错？
需要特别注意什么？
边界情况有哪些？
3. 测试驱动开发（TDD）

核心理念
先写测试，再写代码
测试即文档：测试用例就是最好的使用说明
测试覆盖率不是目标：目标是有意义的测试
测试层次
单元测试（Unit Tests）
测试单个函数/方法的行为
要求快速、独立、可重复
覆盖正常情况和边界情况
集成测试（Integration Tests）
测试多个模块协作
测试 API 接口
测试数据库交互
端到端测试（E2E Tests）
测试完整用户流程
确保关键功能可用
测试实践
每个函数至少要有3个测试用例： 
正常情况（Happy Path）
边界情况（Edge Cases）
错误情况（Error Cases）
测试名称要清晰描述测试内容
使用 Arrange-Act-Assert (AAA) 模式
Mock 外部依赖（API、数据库等）
测试工具栈
Jest: JavaScript 单元测试框架
React Testing Library: React 组件测试
Supertest: API 测试
Prisma Mock: 数据库 Mock
4. 版本控制与进度管理

更新 TodoList 的时机
✅ 功能开发完成
✅ 单元测试全部通过
✅ 集成测试通过
✅ 手动测试验证通过
诚实原则
完成就是完成，不完成就是不完成
不要标记"基本完成"或"差不多完成"
遇到问题要记录，不要掩盖
预估时间不准确要及时调整
Git Commit 规范
feat: 新功能
fix: 修复 bug
test: 添加或修改测试
refactor: 重构代码
docs: 文档更新
style: 代码格式调整
5. 代码审查自查清单

在标记任务完成前，自我审查：

功能性
[ ] 功能是否符合需求？
[ ] 是否处理了所有边界情况？
[ ] 错误处理是否完善？
代码质量
[ ] 代码是否简洁易读？
[ ] 变量和函数命名是否准确？
[ ] 是否有重复代码需要提取？
[ ] 是否有不必要的复杂度？
测试
[ ] 是否写了充足的单元测试？
[ ] 测试是否覆盖了主要场景？
[ ] 所有测试是否通过？
文档
[ ] 是否需要更新 README？
[ ] 是否需要添加代码注释？
[ ] API 是否需要文档？
性能与安全
[ ] 是否有性能问题？
[ ] 是否有安全隐患？
[ ] 数据验证是否充分？
6. 开发节奏

小步快跑
每次只专注一个小任务
完成一个任务，测试通过后，再开始下一个
不要同时开发多个功能
定期回顾
每完成一个大模块，回顾代码
检查是否有可以改进的地方
更新文档和 TodoList
保持专注
不要偏离当前目标
发现新问题记录在 TODO，不要立即去做
保持代码库随时可运行
记住：好的代码不是一次写成的，而是不断重构和改进的结果。
